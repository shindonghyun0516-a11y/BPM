# AGENTS.md

이 문서는 Codex와 멀티 에이전트가 BPM 자동 측정 프로젝트에서 반드시 지켜야 할 핵심 규칙입니다.

자세한 제품/기술/검사 규칙은 `docs/product`와 `docs/rules` 문서를 따른다.

## 1. 역할

Codex는 임의로 기능을 늘리는 아이디어 도구가 아니라, 문서에 맞춰 작은 단위로 구현하고 검증하는 작업자다.

## 2. 문서 우선순위

작업 전 관련 문서를 먼저 확인한다.

1. `README_FOR_PM.md`
2. `WORKFLOW.md`
3. `docs/PROJECT_DECISIONS.md`
4. `docs/product/MVP_SCOPE.md`
5. `docs/product/PRD.md`
6. `docs/product/TRD.md`
7. `docs/rules/*.md`

이 프로젝트는 PM 1명이 운영한다. 작업 방식, Issue, PR, Worktree, 승인 규칙을 먼저 확인한 뒤 제품 범위와 기술 문서를 확인한다.

## 3. 절대 정책

- 기본 측정 시간은 15초다.
- 기본 BPM 범위는 10~500이다.
- 절반/두 배 BPM 후보를 별도 정책으로 강제하지 않는다.
- BPM 결과에는 신뢰도를 표시한다.
- 불안정 결과는 다시 측정 또는 탭 보정을 안내한다.
- 초기 MVP에서는 오디오 저장을 금지한다.
- 초기 MVP에서는 오디오 서버 전송을 금지한다.
- 측정 후 마이크 입력을 중지한다.
- 권한 거부 시 대체 안내를 제공한다.

## 4. 작업 원칙

- Issue 하나는 가능한 한 작은 작업 하나로 유지한다.
- 큰 기능 개발은 Issue 없이 시작하지 않는다.
- Codex는 코드를 작성하기 전에 Issue 초안을 먼저 작성한다.
- PM이 Issue의 목적, 범위, 제외 범위, 완료 기준을 승인한 뒤에만 Issue 생성 또는 구현을 진행한다.
- 기능 구현, 오디오 분석, 마이크 권한, 배포 설정, harness 수정은 Codex App Worktree에서 진행한다.
- README나 문서 초안처럼 위험도가 낮은 작업은 Local 또는 단순 Branch 작업을 허용한다.
- main 브랜치에는 검수되지 않은 변경을 직접 반영하지 않는다.
- V0 harness-check 필수 검사는 build, lint, test, privacy check, BPM result UI check다.
- 초기에는 BPM 정확도 테스트를 자동 차단 조건으로 두지 않고 PM 수동 검수 항목으로 관리한다.
- PR 최종 승인자는 PM 1명으로 한다. 단, 고위험 변경은 바로 병합하지 않는다.
- 코드보다 계약과 상태 모델을 먼저 정한다.
- 오디오 처리, BPM 계산, UI, 개인정보 검증을 분리한다.
- 변경 후에는 관련 테스트 또는 harness 검증을 실행한다.
- 실행하지 못한 검증은 이유를 보고한다.

## 5. 금지 사항

- raw audio를 저장하지 않는다.
- raw audio를 서버로 전송하지 않는다.
- 측정 종료 후 마이크가 계속 켜진 상태를 허용하지 않는다.
- 신뢰도 없는 BPM 결과를 표시하지 않는다.
- 불안정한 결과를 확정값처럼 보여주지 않는다.
- 모바일웹 MVP 범위를 넘어 큰 기능을 임의로 추가하지 않는다.
- 고위험 변경은 Codex review, harness-check 통과, PM 체크리스트 확인 후 병합한다.
- 필요하면 외부 개발자 또는 전문가에게 일회성 리뷰를 요청한다.
- 오디오 서버 전송 가능성 또는 BPM 결과 화면의 신뢰도/후보 BPM 누락은 V0 PR 차단 조건으로 본다.

고위험 변경:

- 마이크 권한 처리 변경
- 오디오 저장 또는 서버 전송 관련 변경
- BPM 계산 알고리즘 변경
- 개인정보/프라이버시 문구 변경
- 배포 설정 변경
- harness-check 우회 또는 삭제
- BPM 결과의 신뢰도/후보 표시 제거
