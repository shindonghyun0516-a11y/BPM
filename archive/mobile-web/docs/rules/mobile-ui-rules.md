# Mobile UI Rules

이 문서는 모바일웹에서 사용자가 BPM 측정을 이해하고 사용할 수 있도록 하는 UI 규칙입니다.

## 1. 기본 방향

이 앱은 모바일 우선으로 만든다. 사용자는 음악을 틀어놓은 상태에서 빠르게 측정하고 결과를 확인해야 한다.

화면은 화려함보다 상태와 다음 행동이 명확해야 한다.

## 2. 필수 화면

### 시작 화면

- 앱이 무엇을 하는지 짧게 설명
- 측정 시작 버튼
- 오디오 저장/전송을 하지 않는다는 안내

### 권한 요청 상태

- 마이크 권한이 필요한 이유 설명
- 브라우저 권한 요청과 연결

### 측정 중 화면

- 남은 시간 또는 진행률
- 현재 측정 중임을 명확히 표시
- 입력이 너무 낮을 때 안내
- 취소 버튼

### 결과 화면

- 추천 BPM 또는 주요 후보
- 후보 BPM 목록
- 신뢰도
- 안정성 상태
- 다시 측정 버튼
- 탭 보정 버튼

### 불안정 결과 화면

- 결과가 불안정한 이유를 쉬운 말로 설명
- 다시 측정 CTA
- 탭 보정 CTA

### 권한 거부 화면

- 마이크 권한이 거부되었음을 안내
- 브라우저 설정에서 권한을 다시 켜는 방법 안내
- 탭 보정 같은 대체 경로 제공

## 3. 결과 표현 규칙

BPM 결과는 너무 확정적으로 표현하지 않는다.

좋은 문구:

- "가장 가능성이 높은 BPM"
- "후보 BPM"
- "신뢰도 높음"
- "측정이 불안정합니다"

피해야 할 문구:

- "정답 BPM"
- "완벽한 BPM"
- "100% 정확"

## 4. 접근성 규칙

- 버튼은 누르기 쉬운 크기여야 한다.
- 텍스트는 모바일에서 읽기 쉬워야 한다.
- 색상만으로 상태를 구분하지 않는다.
- 아이콘 버튼에는 의미를 알 수 있는 label이 있어야 한다.
- 측정 중과 결과 상태는 글자로도 설명한다.

## 5. PM 검수 항목

- 첫 화면에서 무엇을 하는 앱인지 이해되는가
- 측정 중 사용자가 기다릴 이유를 아는가
- 결과 화면에서 신뢰도를 쉽게 이해할 수 있는가
- 불안정 결과가 실패처럼만 보이지 않고 다음 행동을 안내하는가
- 권한 거부 시 사용자가 막히지 않는가

## 6. Harness 후보 검사

나중에 자동 검사로 바꿀 수 있는 항목:

- 주요 CTA 존재 여부 검사
- 결과 화면에 confidence 텍스트 존재 여부 검사
- unstable 상태에 retry/tap correction 버튼 존재 여부 검사
- 모바일 viewport 스크린샷 검사
- 버튼 접근성 label 검사
