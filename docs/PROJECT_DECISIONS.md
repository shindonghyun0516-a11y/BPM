현재 문서 PR은 Issue #1로 보고, 아래 수정사항을 반영한 뒤 첫 문서 베이스라인으로 승인한다.

Issue #1의 목적:

- BPM 자동 측정 앱 프로젝트의 초기 문서 구조 확정
- PM 운영 방식 정리

- MVP 범위 정리
- 오디오 분석, 개인정보, UI, 테스트, harness 규칙 정리

Issue #1 완료 기준:

- `README_FOR_PM.md`에 PM의 다음 행동이 명확히 들어간다.
- `privacy-rules.md`에 마이크 권한 안내 원칙이 보강된다.
- `AGENTS.md`에서 `docs/PROJECT_DECISIONS.md` 또는 운영 결정사항의 우선순위가 앞으로 조정된다.
- `harness-rules.md`에 실제 `harness-check.yml` 생성 시점이 명확히 들어간다.

## 1. GitHub Issue 생성 방식

이 프로젝트에서는 GitHub Issue를 Codex가 초안 작성하고, PM이 승인한 뒤 생성한다.

- Codex는 코드 작성 전에 Issue 초안을 작성한다.
- PM은 목적, 범위, 제외 범위, 완료 기준을 검토한다.
- 승인 후 Codex가 생성하거나 PM이 직접 복사해 생성한다.
- Issue 없이 큰 기능 개발을 시작하지 않는다.

## 2. 작업 방식

이 프로젝트에서는 Codex App Worktree를 기본 작업 방식으로 사용한다.

- 기능 구현, 오디오 분석, 마이크 권한, 배포 설정, harness 수정은 Worktree에서 진행한다.
- README나 문서 초안처럼 위험도가 낮은 작업은 Local/Branch 작업을 허용한다.
- main 브랜치에는 검수되지 않은 변경을 직접 반영하지 않는다.

## 3. harness-check.yml 필수 검사

V0에서는 다음 검사를 필수로 둔다.

- build check
- lint check
- test check
- privacy check
- BPM result UI required check

필수 차단 조건:

- 빌드 실패
- lint 실패
- 테스트 실패
- 오디오 서버 전송 가능성 발견
- BPM 결과 화면에서 신뢰도 또는 후보 BPM 누락

정확도 테스트는 초기에는 자동 차단 조건으로 두지 않고 PM 수동 검수 항목으로 관리한다.

## 4. 배포 플랫폼

초기 배포 플랫폼은 Vercel로 결정한다.

이유:

- 모바일웹 MVP를 빠르게 배포하기 쉽다.
- GitHub와 연결해 PR별 preview 확인이 쉽다.
- Next.js를 사용할 경우 배포 구조가 단순하다.
- 혼자 하는 프로젝트에서 설정 부담이 적다.

단, 프로젝트를 단순 React/Vite 정적 앱으로 만들 경우 Cloudflare Pages도 대안으로 검토한다.

## 5. PR 최종 승인자

이 프로젝트의 PR 최종 승인자는 PM 1명으로 한다.

단, 다음 고위험 변경은 바로 병합하지 않는다.

- 마이크 권한 처리 변경
- 오디오 저장 또는 서버 전송 관련 변경
- BPM 계산 알고리즘 변경
- 개인정보/프라이버시 문구 변경
- 배포 설정 변경
- harness-check 우회 또는 삭제
- BPM 결과의 신뢰도/후보 표시 제거

고위험 변경은 Codex 리뷰, harness-check 통과, PM 체크리스트 확인 후 병합한다.
필요하면 외부 개발자 또는 전문가에게 일회성 리뷰를 요청한다.

## 7. 제품 정책 결정사항

### 7-1. 신뢰도 표시 방식

신뢰도는 초기 MVP에서는 숫자 퍼센트가 아니라 텍스트 등급으로 표시한다.

표시 방식:

- 낮음
- 보통
- 높음

이유:

- 사용자가 퍼센트를 정확한 수치처럼 오해할 수 있다.
- V0에서는 알고리즘 정확도가 아직 검증되지 않았으므로 "87% 신뢰도" 같은 표현은 과도하게 정밀해 보일 수 있다.
- PM/사용자 입장에서는 낮음/보통/높음이 더 이해하기 쉽다.

단, 내부 계산 로직에서는 나중에 숫자 score를 사용할 수 있다. 하지만 사용자 화면에는 우선 낮음/보통/높음으로 표시한다.

### 7-2. 후보 BPM 표시 개수

후보 BPM은 최대 3개까지 표시한다.

기본 구조:

- 추천 BPM 1개
- half-time 후보 1개
- double-time 후보 1개

예시:

- 추천 BPM: 128
- 다른 후보: 64 / 256

표시 원칙:

- 후보가 불확실하면 억지로 3개를 모두 채우지 않는다.
- 가장 중요한 값은 "추천 BPM"으로 크게 보여준다.
- 다른 후보는 보조 정보로 보여준다.
- 사용자가 후보 BPM을 선택할 수 있는 구조는 V1 이후에 검토한다.

### 7-3. 권한 거부 시 안내 정책

마이크 권한을 거부한 경우에는 "탭 보정"을 먼저 안내한다.

우선순위:

1. 탭 보정으로 BPM을 측정할 수 있다는 대안 제공
2. 그 아래에 브라우저 설정에서 마이크 권한을 다시 허용하는 방법 안내

이유:

- 사용자가 권한을 거부했을 때 바로 브라우저 설정 안내를 보여주면 부담스럽다.
- 사용자가 즉시 사용할 수 있는 대안은 탭 보정이다.
- 브라우저 설정 안내는 필요하지만, 1차 CTA가 아니라 보조 안내로 둔다.

권한 거부 화면 예시:

- "마이크 권한이 꺼져 있어 자동 측정을 할 수 없습니다."
- "대신 박자에 맞춰 탭해서 BPM을 측정할 수 있습니다."
- 주요 버튼: "탭으로 측정하기"
- 보조 링크: "마이크 권한 다시 허용하는 방법 보기"
