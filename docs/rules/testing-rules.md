# Testing Rules

이 문서는 BPM 자동 측정 앱을 어떻게 검증할지 정의합니다.

## 1. 테스트 목표

테스트는 단순히 화면이 뜨는지 보는 것이 아니다. 제품 정책이 실제로 지켜지는지 확인해야 한다.

## 2. 필수 검증 항목

- 기본 측정 시간이 15초인지
- BPM 범위가 10~500인지
- 결과에 신뢰도가 포함되는지
- 불안정 결과에 다시 측정과 탭 보정 안내가 있는지
- 권한 거부 시 대체 안내가 있는지
- 마이크 권한 허용 시 측정이 시작되는지
- 마이크 권한 거부 시 사용자가 다음 행동을 알 수 있는지
- 측정 완료 후 마이크가 중지되는지
- 오디오가 저장되지 않는지
- 오디오가 서버로 전송되지 않는지
- 모바일 화면에서 주요 UI가 깨지지 않는지
- 소음이 큰 환경이나 입력이 작은 환경에서 불안정 안내가 나오는지
- 기준 BPM 샘플에서 후보 BPM과 신뢰도가 표시되는지

## 3. 수동 테스트

PM 또는 사람이 직접 확인할 테스트:

1. 모바일 브라우저에서 접속한다.
2. 측정 시작을 누른다.
3. 마이크 권한을 허용한다.
4. 측정 중 진행 상태를 확인한다.
5. 결과에 BPM 후보와 신뢰도가 있는지 확인한다.
6. 다시 측정이 가능한지 확인한다.
7. 탭 보정으로 이동 가능한지 확인한다.
8. 권한 거부 상황에서 안내가 나오는지 확인한다.

## 4. 마이크 권한 테스트

마이크 권한은 사용자가 처음 만나는 중요한 흐름이다.

반드시 확인할 상황:

- 권한 허용: 측정이 시작되고 진행 상태가 보인다.
- 권한 거부: 권한이 필요하다는 안내와 대체 행동이 보인다.
- 권한을 이미 허용한 상태: 다시 측정이 자연스럽게 시작된다.
- 권한을 이미 차단한 상태: 브라우저 설정 안내가 보인다.

## 5. BPM 샘플 테스트

자동 BPM 측정은 기준 샘플로 검증해야 한다.

초기 V0에서는 BPM 정확도 테스트를 자동 차단 조건으로 두지 않는다. 정확도는 PM 수동 검수 항목으로 관리한다.

초기 기준 샘플:

- 80 BPM 샘플
- 120 BPM 샘플
- 160 BPM 샘플

각 샘플에서 확인할 것:

- 후보 BPM 목록이 표시되는가
- 신뢰도가 표시되는가
- 실제 BPM 근처 후보가 포함되는가
- 결과가 불안정하면 재측정 또는 탭 보정 안내가 나오는가

샘플은 실제 음악 파일일 수도 있고, 테스트용으로 만든 단순한 비트 사운드일 수도 있다. 단, 초기 MVP에서는 사용자의 오디오를 서버로 보내는 방식으로 테스트하지 않는다.

## 6. 소음 환경 테스트

실제 사용자는 항상 조용한 환경에서 측정하지 않는다.

확인할 상황:

- 조용한 환경
- 음악 소리가 너무 작은 환경
- 주변 소음이 큰 환경
- 박자가 약하거나 흐릿한 음악

기대 동작:

- 안정적인 경우에는 후보 BPM과 신뢰도를 보여준다.
- 불안정한 경우에는 확정값처럼 보여주지 않는다.
- 불안정한 경우에는 다시 측정 또는 탭 보정을 안내한다.

## 7. 자동 테스트

Codex와 개발자가 작성해야 할 테스트:

- 상태 모델 테스트
- BPM 범위 제한 테스트
- confidence 필드 존재 테스트
- unstable 상태 분기 테스트
- 권한 허용/거부 상태 분기 테스트
- 마이크 stop 호출 테스트
- 오디오 저장/전송 금지 검사

## 8. 모바일 테스트

우선 확인할 환경:

- iOS Safari
- Android Chrome
- 모바일 viewport 브라우저 테스트

모바일에서 확인할 것:

- 버튼을 누르기 쉬운지
- 텍스트가 잘리지 않는지
- 측정 중 상태가 보이는지
- 권한 안내가 이해되는지

## 9. 실패 기준

다음 중 하나라도 해당하면 PR을 통과시키지 않는다.

- 오디오 저장 가능성이 있음
- 오디오 서버 전송 가능성이 있음
- 측정 후 마이크가 꺼지지 않음
- 신뢰도 없는 BPM 결과 표시
- 불안정 결과에 다음 행동 없음
- 기본 정책 15초 또는 10~500 범위 위반
- 권한 거부 시 안내 없음
- 사용자 오디오를 서버로 보내는 코드 또는 동작 존재
- BPM 결과 화면에서 신뢰도 또는 후보 BPM 누락

초기 V0에서는 기준 BPM 샘플의 정확도 부족만으로 PR을 자동 차단하지 않는다. 단, PM 검수에서 리스크로 기록한다.

## 10. PM 검수 항목

- 테스트 결과가 PR 설명에 적혀 있는가
- 실행하지 못한 테스트와 이유가 적혀 있는가
- 모바일 화면 확인이 포함되어 있는가
- 개인정보 관련 검증이 포함되어 있는가
- 기준 BPM 샘플 또는 그 대체 테스트가 포함되어 있는가
- 소음/작은 입력 환경에서의 동작 확인이 포함되어 있는가

## 11. 스윙 계열 BPM 후보 테스트

이 프로젝트의 1차 타깃은 스윙 재즈, 리듬앤블루스, 뉴올리언스, 빅밴드, 스윙 계열 음원이다. 따라서 BPM 하나를 정답처럼 맞히는 것보다 실용적인 후보 BPM이 신뢰도와 함께 표시되는지 확인한다.

### 테스트 그룹

| 그룹 | 대상 | 기대 결과 | 병합 보류 기준 |
| --- | --- | --- | --- |
| A. 기준 정규 비트 | metronome 90/120, 단순 4/4 드럼 루프 | 기존처럼 추천 BPM 또는 안정 후보 표시 | 기존에 잡히던 정규 비트가 불안정으로 바뀜 |
| B. 2&4 강세 | 스윙 재즈, ride/piano/bass 중심 | 참고 BPM 후보 또는 half/double 후보 표시 | 충분한 입력인데 후보가 항상 없음 |
| C. 1&3 강세 | R&B, 뉴올리언스, shuffle/backbeat | 참고 BPM 후보 또는 half/double 후보 표시 | half/double 후보가 사라짐 |
| D. 빅밴드 | 브라스 어택이 많고 강세가 큰 곡 | 후보가 갈리면 참고 BPM 후보로 조심스럽게 표시 | 갈린 후보를 신뢰도 높음으로 확정 |
| E. 피아노 솔로/루바토 | 자유로운 템포, 독주 | 참고 후보 또는 불안정 안내 | 후보 부족인데 BPM을 억지로 표시 |

### QA 기록 양식

| 그룹 | 곡명 | 알려진 BPM 또는 체감 BPM | 화면 결과 | 추천 BPM 또는 참고 BPM 후보 | 후보 BPM | 신뢰도 | 불안정 사유 | PM 판단 | 비고 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| A/B/C/D/E |  |  | 추천/참고/불안정 |  |  | 낮음/보통/높음 |  | 통과/수정 필요/보류 |  |

### 판단 원칙

- A 그룹이 나빠지면 병합을 보류한다.
- B/C/D 그룹은 단일 정답보다 후보군에 실용 BPM 또는 half/double 후보가 들어있는지 본다.
- E 그룹은 무조건 성공을 요구하지 않는다.
- 무음, 입력 부족, onset 부족에서는 BPM 후보를 표시하지 않는다.
- 참고 BPM 후보는 정답처럼 보이면 안 된다.
