# README FOR PM

이 문서는 개발자가 아닌 PM이 BPM 자동 측정 프로젝트를 운영하기 위한 쉬운 안내서입니다.

## 이 프로젝트를 한 문장으로 설명하면

사용자가 음악을 틀어놓고 측정을 시작하면, 모바일 기기의 마이크로 BPM을 자동 추정하고 후보 BPM과 신뢰도를 보여주는 앱입니다.

## PM이 꼭 기억해야 할 기준

- 첫 버전은 모바일웹 MVP입니다.
- 사용자가 버튼을 박자에 맞춰 누르는 Tap Tempo가 아니라 자동 측정이 핵심입니다.
- BPM은 하나의 정답처럼 보이면 안 됩니다. 후보와 신뢰도가 같이 보여야 합니다.
- 불안정한 측정은 솔직하게 불안정하다고 보여주고, 다시 측정 또는 탭 보정을 안내해야 합니다.
- 초기 MVP에서는 오디오 저장과 서버 전송이 금지입니다.
- 측정이 끝나면 마이크가 꺼져야 합니다.

## PM이 자주 보게 될 용어

### Issue

해야 할 일을 적어두는 작업 카드입니다. 예를 들어 "마이크 권한 요청 화면 만들기"가 하나의 Issue가 될 수 있습니다.

### Branch

작업을 따로 해보는 임시 작업선입니다. 완성 전까지는 main에 바로 들어가지 않습니다.

### Worktree

Codex가 특정 Issue를 안전하게 작업할 수 있는 별도 작업 공간입니다. 이 프로젝트에서는 기능 구현, 오디오 분석, 마이크 권한, 배포 설정, harness 수정 같은 중요한 작업은 Worktree를 기본으로 사용합니다.

README나 문서 초안처럼 위험도가 낮은 작업은 Local 또는 단순 Branch 작업을 허용합니다. main 브랜치에는 검수되지 않은 변경을 직접 반영하지 않습니다.

### PR

Pull Request의 줄임말입니다. 작업한 내용을 main에 넣기 전에 검수해달라는 요청입니다.

### Harness

자동 검사입니다. 예를 들어 "측정 시간이 15초인지", "오디오를 서버로 보내지 않는지", "측정 후 마이크를 끄는지" 같은 정책을 검사합니다.

### Deploy

사용자가 접속할 수 있는 실제 웹 주소에 배포하는 일입니다.

## PM 운영 흐름

1. 요구사항을 Issue로 적습니다.
2. Planner agent가 범위와 리스크를 정리합니다.
3. Codex가 Branch와 Worktree에서 작업합니다.
4. Harness가 자동 검사를 실행합니다.
5. PR이 생성됩니다.
6. Codex review와 사람 검수를 진행합니다.
7. GitHub Actions 검사가 통과하면 merge합니다.
8. main에 merge되면 deploy workflow가 배포합니다.

## PM 빠른 체크리스트

- [ ] 현재 작업이 어떤 Issue와 연결되어 있는가?
- [ ] Codex가 코드를 수정하기 전에 계획을 먼저 제시했는가?
- [ ] BPM 결과에 추천 BPM, 후보 BPM, 신뢰도가 포함되는가?
- [ ] 마이크 권한 안내가 충분한가?
- [ ] 오디오를 저장하거나 서버로 전송하지 않는가?
- [ ] 권한 거부 시 탭 보정 대안이 있는가?
- [ ] 테스트 또는 수동 검수 항목이 정리되어 있는가?
- [ ] 내가 승인하기 전까지 main에 병합하지 않는가?

## PM이 PR에서 확인할 것

- 이번 PR이 Issue 범위만 다루는지
- 제품 정책을 바꾸지 않았는지
- BPM 결과에 신뢰도가 표시되는지
- 불안정 결과 안내가 있는지
- 오디오 저장/서버 전송이 없는지
- 측정 후 마이크가 중지되는지
- 권한 거부 시 사용자가 다음 행동을 알 수 있는지
- Harness 결과가 통과했는지

## GitHub Actions harness 확인 방법

Issue #3부터 PR 또는 main push가 발생하면 GitHub Actions에서 `harness-check`가 실행됩니다.

PM 확인 순서:

1. GitHub 저장소에서 PR 화면을 엽니다.
2. PR 하단의 checks 영역을 확인합니다.
3. `harness-check`가 보이는지 확인합니다.
4. `Build`, `Lint`, `Test`, `Privacy check`, `BPM result UI check` 단계가 통과했는지 확인합니다.
5. 실패한 단계가 있으면 그 PR은 바로 병합하지 않습니다.
6. 실패 원인을 Codex에게 요약하게 하고 수정 후 다시 검사합니다.

Issue #3에서 자동 차단하는 항목:

- build 실패
- lint 실패
- test 실패
- 오디오 파일 업로드, 마이크 입력 서버 전송, recording upload API 가능성 발견
- 녹음 파일 저장 가능성 발견

Issue #5부터 추가로 자동 차단하는 항목:

- 추천 BPM UI 누락
- 후보 BPM UI 누락
- 신뢰도 UI 또는 `낮음 / 보통 / 높음` 라벨 누락
- 다시 측정 버튼 누락
- 불안정 결과 안내 누락
- 권한 거부 화면의 탭 보정 예정 안내 누락
- 실제 탭 측정 기능처럼 보이는 금지 문구 발견

Issue #3에서 PM 확인으로 분리하는 항목:

- 저장 또는 업로드로 이어질 수 있지만 자동 실패로 단정하기 어려운 recording 관련 경고

아직 자동 차단하지 않는 항목:

- BPM 정확도 검사
- 모바일 브라우저 실제 사용성 검사

BPM 결과 UI 필수 검사는 Issue #5부터 자동 검사에 포함합니다. BPM 정확도는 V0에서는 PM 수동 검수 항목으로 관리합니다.

## PM이 결정해야 하는 항목

- 모바일앱 확장을 언제 시작할지
- 측정 실패 또는 소음이 큰 환경에서 어떤 문구를 보여줄지
- 기준 BPM 샘플을 실제 음악으로 준비할지, 테스트용 단순 비트로 만들지

## 현재 확정된 운영 결정

자세한 내용은 [docs/PROJECT_DECISIONS.md](docs/PROJECT_DECISIONS.md)에 기록합니다.

- GitHub Issue는 Codex가 초안을 작성하고 PM이 승인한 뒤 생성합니다.
- 기본 작업 방식은 Codex App Worktree입니다.
- 낮은 위험의 문서 작업은 Local 또는 단순 Branch 작업을 허용합니다.
- Issue #3 `harness-check.yml` 필수 검사는 build, lint, test, privacy check입니다.
- BPM result UI check는 Issue #5부터 필수 검사로 활성화합니다.
- 초기에는 BPM 정확도 테스트를 자동 차단 조건으로 두지 않고 PM 수동 검수 항목으로 관리합니다.
- 모바일웹 MVP 초기 배포 플랫폼은 Vercel로 결정합니다.
- GitHub main 브랜치 병합 후 Vercel production 배포를 목표로 합니다.
- PR 단계에서는 Vercel preview 배포를 확인할 수 있게 합니다.
- 배포 전에는 `harness-check.yml`이 통과해야 합니다.
- Cloudflare Pages는 기본 배포 플랫폼이 아니라 대안으로만 문서에 기록합니다. 단순 React/Vite 정적 앱으로 만들 경우에만 검토합니다.
- PR 최종 승인자는 PM 1명으로 합니다.
- 고위험 변경은 바로 병합하지 않습니다. Codex review, harness-check 통과, PM 체크리스트 확인 후 병합합니다.
- 필요하면 외부 개발자 또는 전문가에게 일회성 리뷰를 요청합니다.
- Issue #1은 아래 문서 보강 후 첫 문서 베이스라인으로 승인합니다.
- 다음 작업 순서는 Issue #2 Next.js 초기화, Issue #3 harness-check 기본 검사 구성, Issue #4 마이크 기반 BPM 측정 프로토타입입니다.
- 초기 MVP의 신뢰도는 사용자 화면에서 낮음/보통/높음으로 표시합니다.
- 후보 BPM은 최대 3개까지 표시합니다. 추천 BPM 1개를 크게 보여주고, half-time 후보와 double-time 후보는 보조 정보로만 보여줍니다.
- 권한 거부 시에는 탭 보정을 먼저 안내하고, 브라우저 마이크 권한 설정 안내는 보조 안내로 둡니다.

- 오디오 저장 또는 서버 전송 관련 변경

- 개인정보/프라이버시 문구 변경
- 배포 설정 변경

- BPM 결과의 신뢰도/후보 표시 제거
